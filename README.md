# Mint

Mint lets you create "roles" consisting of flexible IAM policy templates associated with Tigris access keys.
We use macaroons for authorization, holder-of-key proof-of-possession for authentication, and third-party caveats for attestation.
Policy templates support `{{caveat.X}}` expressions resolved from caveats on the macaroon itself, and a derived `{{mint.expiry}}` expression that resolves to an expiry timestamp. Deployment constants are written into the template as plain literals.

Mint extends the simplified [Tigris IAM](https://www.tigrisdata.com/docs/iam/) model, with the ability to exchange long-lived "service tokens" for temporary, limited-privilege credentials defined by policy templates. Think of this as _roughly_ analogous to a lightweight macaroon-aware STS (but don't quote me on that).

This ensures admin credentials remain tightly controlled _within_ the mint process, while exposing API-based minting of least-privilege credentials for trusted clients.

A simple example policy is shown below. Note the `{{...}}` template expressions.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::{{caveat.bucket}}/{{caveat.sub}}/*"],
      "Condition": {
        "DateLessThan": {"aws:CurrentTime": "{{mint.expiry}}"}
      }
    }
  ]
}
```

* `{{caveat.bucket}}` - a value the caller proposes at exchange, vouched by an attestation authority (*attested*)
* `{{caveat.sub}}` - the client identifier mint stamps from the macaroon (*issuer-stamped*)
* `{{mint.expiry}}` - policy expiration (*mint-computed*)

A policy is generated by replacing these expressions in the template.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::my-bucket/01ARZ3NDEKTSV4RRFFQ69G5FAV/*"],
      "Condition": {
        "DateLessThan": {"aws:CurrentTime": "2026-06-18T20:24:34+00:00"}
      }
    }
  ]
}
```

Every `{{caveat.X}}` value is MAC-verified on the credential. Its *provenance* is derived from the name — there is nothing to declare:

* *issuer-stamped* — a reserved name (e.g. `sub`) mint sets from the principal
* *attested* — any other name: the caller proposes it at exchange (`--caveat X=V`) and an attestation authority vouches it before it is baked in

A role binding any *attested* caveat carries a third-party caveat and exchanges in two steps (intermediate → discharge → finalize); a role binding only issuer-stamped caveats needs no authority and exchanges in one.

## Getting Started

Initial configuration, admin key management and running the mint server -
```bash
cp examples/mint-demo.toml ./mint-demo.toml   # then edit store.bucket to a bucket you own
export MINT_CONFIG=./mint-demo.toml

# Note: mint-demo.toml defines 2 roles and references the following templates by default -
# * ./examples/demo_roles/demo.json
# * ./examples/demo_roles/demo-attested.json

# Then either (A) or (B) below -

# (A) Tigris admin credentials in 1Password
cp examples/mint-demo.env ./mint-demo.env    # then edit to match your vault and path
op run --env-file ./mint-demo.env -- ./target/debug/mint serve

# (B) Tigris admin credentials directly exported
export AWS_ACCESS_KEY_ID=<KEY_ID>
export AWS_SECRET_ACCESS_KEY=<SECRET_KEY>
./target/debug/mint serve
```

With `mint serve` running we can interact with it via the mint cli in a new terminal. 
Note that the mint server by default runs with a demo authentication service available via `auth.sock` locally.

```bash
export MINT_CONFIG=./mint-demo.toml

# Login via mint cli
./target/debug/mint login

# "Seal" the example policy templates (to prevent them being tampered with)
./target/debug/mint seal

# Display the <INVITE> for enrolling a new client
./target/debug/mint invite
```

The mint cli includes a demonstration `client` sub-cmd to allow the enrollment flow to be exercised.

```bash
# Client begins the enrollment process, providing the <INVITE> from earlier -
./target/debug/mint client enroll demo_client <INVITE>

# The operator can then approve the enrollment request -
./target/debug/mint enroll list
./target/debug/mint enroll approve demo_client

# The client fingerprint should be verified out-of-band -
./target/debug/mint client fingerprint
```

Once a client has successfully enrolled with mint it can exchange its credentials for per-role long-lived "service tokens". The client can then "assume-role" swapping a service token for short-lived Tigris/S3 credentials associated with an IAM policy built from the associated template.

```bash
# List the available roles (these are what we "sealed" earlier) -
./target/debug/mint role list

# Exchange for a long-lived service token for the "demo" role. It is
# issuer-only (scoped to your own {{caveat.sub}}), so it takes no values
# and needs no attestation authority -
./target/debug/mint client exchange demo

# Assume this demo role to obtain short-lived Tigris access keys -
./target/debug/mint client assume-role demo
```

For a more complex example the `demo-attested` role binds values a third party must vouch for. By default mint runs with a demo attestation service available via `attest.sock` locally.

The template for the `demo-attested` role substitutes two *attested* `{{caveat.X}}` values — the caller proposes each and the authority vouches it before it is baked in:

* `{{caveat.bucket}}` - the bucket to scope to
* `{{caveat.project}}` - a role-specific value

```bash
# Exchange for the role-specific service token, proposing the bucket and
# project values the authority vouches at finalize -
./target/debug/mint client exchange demo-attested --caveat bucket=my-bucket --caveat project=apollo

# Assume the role to obtain short-lived Tigris access keys
./target/debug/mint client assume-role demo-attested
```
